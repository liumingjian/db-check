# 巡检报告模板设计要点

## 1. 目标

本规范用于指导其他工程师设计新的巡检报告模板，确保不同数据库、不同报告类型在内容组织、表格表现、数据口径和可落地性上保持一致。

设计模板时，必须满足以下原则：

- 报告结构稳定，章节层次清晰。
- 内容来源明确，不能混淆采集事实、分析结论和人工补充信息。
- 不造数据，不隐式补值，不用模板文案掩盖采集缺口。
- 模板首先服务于交付可读性，其次才是视觉表现。

## 2. 内容分层

所有报告内容统一分为三层：

1. 固定文案层
- 章节标题
- 表头
- 等级定义
- 说明文案

2. 动态数据层
- 采集结果 `result`
- 分析结果 `summary`
- 人工补充信息 `meta`

3. 展现模型层
- 将固定文案和动态数据装配成统一的报告视图模型
- 模板渲染只能消费视图模型，不能直接拼原始 JSON

## 3. 推荐章节结构

通用巡检报告建议采用以下结构：

1. 文档控制
- 文档信息
- 修改记录
- 审阅记录

2. 巡检总结
- 巡检告警定义
- 巡检范围
- 综合健康评估
- 风险发现与整改建议
- 巡检结论

3. 巡检明细
- 系统指标
- 数据库基础信息
- 数据库性能检查
- 备份与恢复
- 其他数据库专属章节

说明：
- `文档控制` 和 `巡检总结` 是强稳定章节，跨模板尽量保持一致。
- `巡检明细` 按数据库类型扩展，但章节命名和组织方式要保持统一风格。

## 4. 数据来源口径

每个输出字段都必须能追溯来源，并归入以下一种类型：

- `collected`
  - 直接来自采集结果
- `derived`
  - 来自规则分析、聚合计算或汇总推导
- `external`
  - 来自人工补充元数据
- `missing`
  - 当前未采集到，模板中显式展示“未采集”
- `na`
  - 当前场景不适用，模板中显式展示“不适用”

禁止以下行为：
- 因字段缺失而静默不展示对应章节
- 用默认文案伪装成真实结果
- 用假值填满模板

## 5. 表格设计规范

报告中的表格只允许两种模式。

### 5.1 单行参数表

适用场景：
- 参数/属性型信息
- 列数少
- 强调“名称-值”对应关系

统一规则：
- 列名固定为：`参数名称`、`当前值`
- 不增加字段说明
- 不使用简称列名

典型场景：
- 实例基础信息
- 数据库容量
- 数据库用户
- 连接数检查
- 备份策略

### 5.2 多行记录表

适用场景：
- Top N 列表
- 锁、SQL、对象、索引等明细记录
- 多行多列

多行记录表进一步分为两类。

#### 5.2.1 普通多行记录表

适用条件：
- 列数较少
- 表头全称不会明显影响可读性

规则：
- 直接使用全称表头
- 不增加字段说明

典型场景：
- 系统指标
- 没有主键或唯一键的表
- 非 Innodb 引擎的数据对象
- 单张表字段个数大于 50 的对象
- 全表扫描的 SQL top10
- 未使用的索引
- 自增值使用率 top10

#### 5.2.2 宽表紧凑记录表

适用条件：
- 列数较多
- 列名较长
- 全称表头会导致版面拥挤或换行难看

规则：
- 表头使用短字段名
- 表格下方增加“字段说明”
- 字段说明格式统一为：

```md
字段说明：

- `field_a`: 含义说明
- `field_b`: 含义说明
```

典型场景：
- 元数据锁信息
- 占用空间 top 10 的表
- 慢 SQL top10
- 冗余索引
- 其他列较多且名称较长的对象明细表

## 6. 不同章节的表格风格约束

### 6.1 文档控制

固定使用全称表头。

原因：
- 这部分内容稳定
- 所有报告都通用
- 不存在字段过长导致的展示问题

### 6.2 巡检总结

固定使用全称表头。

原因：
- 这是面向读报告的人首先阅读的部分
- 强调业务可读性，不强调技术字段压缩
- 需要尽量少的术语缩写

### 6.3 巡检明细

仅在“宽表”场景下允许使用简称表头加字段说明。

原因：
- 这部分偏技术细节
- 存在列多、字段长的情况
- 允许为版面可读性做紧凑化处理

## 7. 文案风格要求

### 7.1 标题

- 使用业务可理解标题
- 避免仅使用内部字段名
- 层级清晰，编号稳定

### 7.2 表头

- 面向业务阅读时优先使用中文全称
- 面向技术细节且列较多时可使用短字段名
- 简称必须可通过字段说明还原

### 7.3 说明文案

- 简短、直接、可执行
- 避免空泛描述
- 避免“可能、大概、也许”这类弱表述

### 7.4 风险表述

风险项至少要包含：
- 风险等级
- 检查维度
- 风险描述
- 影响分析
- 整改建议

## 8. 缺失值和不适用处理

### 8.1 未采集

当底层没有数据时：
- 章节仍保留
- 表格或内容显示“未采集”
- 状态标记为 `missing`

### 8.2 不适用

当当前实例或场景本身不适用时：
- 章节仍保留
- 明确展示“不适用”原因
- 状态标记为 `na`

例如：
- 单机实例未配置复制，则主从状态章节显示“不适用”

## 9. 模板设计流程

设计其他模板时，建议按以下顺序进行：

1. 先确定章节结构
- 哪些章节属于固定通用章节
- 哪些章节属于数据库专属章节

2. 再梳理字段来源
- 哪些字段来自 `result`
- 哪些字段来自 `summary`
- 哪些字段来自 `meta`

3. 再设计表格表现
- 是单行参数表还是多行记录表
- 多行记录表是否属于宽表

4. 最后做模板美化
- 只有在内容结构和数据口径稳定后，才进行 Word 样式设计

## 10. 评审检查清单

设计一个新模板时，至少检查以下事项：

- 章节结构是否稳定
- 表格是否只使用规定的两类模式
- 文档控制和巡检总结是否保持全称表头
- 巡检明细中的宽表是否增加字段说明
- 每个输出字段是否能追溯来源
- 缺失值是否显式暴露
- 不适用场景是否明确说明
- 是否存在假值或静默补值
- 风险表述是否完整
- 模板是否便于后续 Word 套版实现

## 11. 当前模板的落地结论

基于当前 MySQL 模板实践，推荐将以下规则作为默认标准：

- 文档控制：固定全称表头
- 巡检总结：固定全称表头
- 巡检明细：
  - 窄表保留全称表头
  - 宽表使用简称 + 字段说明
- 单行参数型内容：统一使用 `参数名称 / 当前值`
- 缺口内容：保留章节并显式标记 `missing` 或 `na`

该规范可直接作为其他数据库模板设计的基线。
