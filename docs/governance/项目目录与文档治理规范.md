# 项目目录与文档治理规范

本文档说明当前仓库为什么按现在的目录方式组织，以及从企业级项目规范角度，哪些目录已经合理，哪些需要约束。

## 1. 总体原则

目录治理采用三条主线：

1. 代码按职责分层
- 采集、分析、报告、规则、contracts、测试分开

2. 文档按受众和作用分层
- 项目入口文档放根目录
- 全局说明文档放 `docs/`
- 模块内局部说明保留在模块目录
- 程序运行时资产和说明性文档分开

3. 入口与构建产物收敛
- 客户可执行入口由 `cmd/` 定义，由 `dist/` 承载发布结果
- 开发辅助脚本统一放 `scripts/`
- 根目录不再散落二进制和 wrapper 脚本

## 2. 当前推荐目录视图

```text
.
├── README.md                    # 项目入口
├── AGENTS.md                    # Agent 规则
├── collector/                   # Go 采集端
├── analyzer/                    # Python 分析端
├── reporter/                    # Python 报告端
│   └── templates/               # 运行时模板资产
├── contracts/                   # machine-readable 契约资产
├── rules/                       # 检查规则
├── tests/                       # 单测、集成测试、e2e
├── scripts/                     # 初始化与辅助脚本
├── dist/                        # 多平台发布包
└── docs/                        # 全局文档中心
```

## 3. docs 为什么要集中管理

把全局文档集中到 `docs/` 有 4 个直接收益：

1. 新同学可以从一个地方找到架构、规范、设计、模板来源和治理说明。
2. 架构师能直接从 `docs/` 建立对系统的全局认知，不需要在 `contracts/`、`reporter/templates/`、根目录之间来回跳转。
3. 运行时目录会更干净，不会把说明文档和机器资产混在一起。
4. 后续如果接入文档站点、Wiki 同步或 CI 文档发布，`docs/` 可以直接作为统一入口。

## 4. 哪些文档应该进 docs

适合放到 `docs/` 的文档：

- 架构说明
- 业务流程说明
- 需求与 PRD
- 开发规范
- 报告设计规范
- 参考模板和历史模板
- 项目治理规则

不建议放到 `docs/` 的内容：

- 程序运行时需要直接读取的模板资产
- contracts 的 schema 和 sample JSON
- rules 本体
- 测试运行产物
- 紧贴某个模块的局部 README

## 5. 哪些目录当前已经合理

### 5.1 scripts/

合理。辅助脚本集中在 `scripts/`，更符合开源社区常见做法；脚本不会与正式发布物目录混用。

### 5.2 dist/

合理。`dist/` 作为发布物目录，专门承载多平台二进制、runtime、assets 和 Quick Start，职责明确。

### 5.3 collector/

合理。Go 采集端单独分层，`cmd/` 和 `internal/` 结构清晰，符合企业项目常见组织方式。

### 5.4 analyzer/

合理。CLI、evaluator、validator、model、writer 的职责清晰，便于扩规则和扩校验链路。

### 5.5 reporter/

总体合理，但要严格区分：
- `templates/` 只保留运行时模板资产
- 设计文档不要继续堆在 `templates/` 下

### 5.6 contracts/

合理，但应尽量保持 machine-readable 导向：
- schema
- sample json
- 未来可增加 contract fixtures

说明性 markdown 不应继续留在这里。

### 5.7 tests/

合理。按 analyzer / collector / integration / e2e / reporter 分层清楚，便于不同层级验证。

`tests/e2e/README.md` 这类局部说明建议继续保留在模块旁边，不必挪到 `docs/`。

## 6. 为什么有些语言相关文件仍应保留在根目录

### 6.1 go.mod / go.sum

应继续保留在根目录。

原因：
- 当前 Go module 根就是仓库根
- `collector` 代码 import path 使用 `dbcheck/...`
- 如果把 `go.mod`/`go.sum` 挪到 `collector/`，会引发 import path、构建脚本和测试入口的系统性重构
- 这不是“整理目录”，而是“改变模块边界”

### 6.2 requirements.txt

也应继续保留在根目录。

原因：
- 当前 Python 依赖同时服务于 analyzer、reporter、tasks、tests
- 它是仓库级运行环境定义，不是某个子模块私有依赖
- 对小白来说，根目录初始化 `.venv + requirements.txt` 更直观

## 7. 当前已执行的目录治理动作

1. 根目录的全局 markdown 已迁入 `docs/`
2. `contracts/` 下的说明性 markdown 已迁入 `docs/`
3. `reporter/templates/` 下的说明性 markdown 已迁入 `docs/`

5. `bin/` 作为本地编译产物目录保留；不再承载脚本入口
6. 空目录 `reporter/template/` 已删除
7. 已新增 `.gitignore` 并清理仓库噪声文件

## 8. 对不同角色分别有什么帮助

### 8.1 小白

从 `README.md -> docs/README.md -> docs/architecture/业务全景与实现流程.md` 就能建立主链路认知。

### 8.2 架构师

可以快速聚焦：
- `docs/architecture/`
- `docs/specs/`
- `docs/governance/`

### 8.3 开发者

可以按工作方向进入：
- 规则与 contracts：`rules/`、`contracts/`
- 报告实现：`reporter/`
- 采集实现：`collector/`
- 回归验证：`tests/`

## 9. 当前不建议进一步大改的部分

当前不建议把 `collector/analyzer/reporter` 再抽象成更多层级目录，原因是：

- 代码规模还没有大到需要更细粒度的工程拆分
- 当前职责边界已经比较清楚
- 过早引入更多目录层级，只会提高新人的理解成本

当前最优先的规范化重点不是代码目录再拆分，而是：
- 文档入口清晰
- 运行时资产与说明文档分离
- 测试与交付路径一致
- 仓库根目录尽量只保留仓库级配置与入口
