# DB-Check 开发规范

## 1. 文档元信息与版本策略

| 字段 | 值 |
|---|---|
| 文档名称 | DB-Check 开发规范 |
| 文档版本 | v2.0.0 |
| 适用范围 | MySQL MVP（采集-分析-报告三段链路） |
| 读者 | 研发、测试、DevOps |
| 状态 | 可执行（可直接指导开发） |
| 最后更新 | 2026-03-05 |

### 1.1 规范优先级
发生冲突时按以下优先级解释：
1. `contracts/schemas/*.json` + `docs/specs/冻结契约说明.md`
2. 本文档 `docs/specs/开发规范.md`
3. `docs/specs/db-check-contracts-prd.md`

### 1.2 版本与兼容策略
1. v2.0 仅支持三层契约主链路：`manifest/result/summary`。
2. 旧链路（单 JSON + Reporter 判定）在 v2.0 已移除。
3. 旧数据不兼容，必须转换为三层契约后才能进入 v2.0 链路。
4. 未来版本可扩展数据库类型，但本版接口定义只覆盖 MySQL。

### 1.3 时间字段策略（本版显式例外）
1. contracts 仍是主规范，时间字段应使用 RFC3339。
2. 业务解释层采用“本地时区优先”规则：
- 报表展示以采集机本地时区为准。
- 文件命名以采集机本地时区为准。
- 解析冲突时优先本地时区语义。
3. 该例外需在后续版本持续审计，必要时回归严格时区策略。

## 2. 目标与非目标

### 2.1 目标（必须达成）
1. 建立可回放、可审计的三段执行链路。
2. 明确组件职责边界：Collector 采集、Analyzer 判定、Reporter 渲染。
3. 强制 contracts 校验门禁，保证数据契约稳定。
4. 提供可落地的工程结构、接口、测试和 CI 标准。

### 2.2 非目标（本版不做）
1. 不实现 Oracle 运行路径。
2. 不定义 Oracle 参数接口。
3. 不引入 Web UI、在线服务化调度、实时告警。
4. 不在 Reporter 中执行规则判定。

## 3. 技术选型与运行时依赖

### 3.1 组件与语言
| 组件 | 技术栈 | 说明 |
|---|---|---|
| Collector | Go 1.21+ | 单二进制、离线部署 |
| Analyzer | Python 3.8+ | 契约校验与规则判定 |
| Reporter | Python 3.8+ | 报告渲染 |

### 3.2 关键依赖（锁定）
1. Collector
- `github.com/go-sql-driver/mysql`
- `github.com/shirou/gopsutil`
- `golang.org/x/crypto/ssh`
2. Analyzer/Reporter
- `jsonschema`（CI 严格模式必须安装）
- `python-docx`

### 3.3 运行约束
1. Collector 必须支持离线场景，不依赖外部服务。
2. Analyzer/Reporter 读取本地文件，不依赖数据库连接。
3. 所有组件必须支持非交互批处理执行。

## 4. 总体架构与职责边界

### 4.1 架构总览
```text
Collector
  -> collector.log
  -> manifest.json
  -> result.json
            |
            v
         Analyzer
  (manifest + result + rule)
            |
            v
        summary.json
            |
            v
         Reporter
 (result + summary + template)
            |
            v
         report.docx
```

### 4.2 组件职责（必须）
1. Collector
- 负责采集与产物编排。
- 不负责风险等级判定。
2. Analyzer
- 唯一风险判定责任方。
- 负责缺失语义映射与 `summary` 生成。
3. Reporter
- 仅负责渲染展示。
- 禁止执行阈值计算、规则判定。

### 4.3 运行目录规范（强制）
```text
runs/<run_id>/
  collector.log
  manifest.json
  result.json
  summary.json
  report.docx
```

## 5. 数据契约与数据模型

### 5.1 契约文件定义
| 文件 | 角色 | 关键要求 |
|---|---|---|
| `manifest.json` | 执行态 | 记录退出码、模块状态、产物索引 |
| `result.json` | 事实层 | 只存事实，不含执行态/分析结论 |
| `summary.json` | 分析层 | 风险等级、计数、缺失映射、失败摘要 |
| `rule.json` | 规则层 | 按 `rule.schema.json` 定义判定规则 |

### 5.2 最小关键字段（必须）
1. `manifest.json`
- `schema_version`, `run_id`, `db_type`, `start_time`, `end_time`
- `exit_code`, `overall_status`, `module_stats`, `artifacts`
2. `result.json`
- `meta`, `collect_config`, `collect_window`, `os`, `db`
3. `summary.json`
- `schema_version`, `run_id`, `rule_version`, `generated_at`, `overall_risk`
- `counts`, `abnormal_items`, `unevaluated_items`, `na_items`
- 当 `manifest.exit_code in {20,30}` 时必须有 `failure`

### 5.3 跨文件一致性规则（必须）
1. `summary.run_id == manifest.run_id`
2. `result.meta.db_type == manifest.db_type == "mysql"`
3. `summary.rule_version == rule.rule_meta.rule_version`
4. `summary.counts.total_checks == normal + warning + critical + unevaluated + not_applicable`
5. `summary.counts.unevaluated == len(summary.unevaluated_items)`
6. `summary.counts.not_applicable == len(summary.na_items)`

### 5.4 缺失语义映射（冻结）
| 语义 | summary 归类 |
|---|---|
| `failed` | `unevaluated_items` |
| `skipped` | `unevaluated_items` |
| `not_applicable` | `na_items` |

### 5.5 事实层硬约束（必须）
1. `result.json` 禁止出现执行态字段（如 `exit_code`、`module_stats`）。
2. 原始值必须机器可读；禁止单位拼接字符串（如 `"419 ms"`）。
3. 样本时间必须落在 `collect_window` 范围内。

### 5.6 规则文件组织（固定）
1. 路径固定：`rules/mysql/rule.json`
2. 版本来自：`rule_meta.rule_version`
3. 本版不支持多文件聚合规则输入。

## 6. CLI 接口规范

## 6.1 Collector CLI（MySQL）

### 6.1.1 基本语法
```bash
db-collector --db-type mysql [连接参数] [采集参数]
```

### 6.1.2 公共参数
| 参数 | 短参数 | 说明 | 默认值 |
|---|---|---|---|
| `--db-type` | `-t` | 数据库类型，仅允许 `mysql` | 必填 |
| `--db-host` | `-h` | 数据库主机地址（远程模式） | 远程模式必填 |
| `--db-port` | `-P` | MySQL 端口 | `3306` |
| `--db-username` | `-u` | 数据库用户名 | 必填（非 `--os-only`） |
| `--db-password` | `-p` | 数据库密码 | 必填（非 `--os-only`） |
| `--dbname` | `-d` | 数据库名 | 必填（非 `--os-only`） |
| `--local` |  | 本地采集模式 | `false` |
| `--os-only` |  | 仅采集 OS（旁路） | `false` |
| `--os-skip` |  | 跳过 OS 指标采集 | `false` |
| `--output-dir` | `-o` | 输出目录 | `./output/` |
| `--log-path` |  | 日志目录或日志文件路径 | `./logs/` |
| `--log-level` |  | 日志级别 | `INFO` |
| `--sql-timeout` |  | 单条 SQL 超时（秒） | `180` |
| `--top-n` | `-n` | Top N 返回条数 | `20` |
| `--version` | `-v` | 显示版本信息 | - |
| `--help` |  | 显示帮助信息 | - |

### 6.1.3 SSH / 远程 OS 参数

远程 OS 采集当前基于 Linux SSH 执行。

| 参数 | 说明 | 默认值 |
|---|---|---|
| `--os-host` | OS 主机地址 | 与 `--db-host` 相同 |
| `--os-port` | SSH 端口 | `22` |
| `--os-username` | SSH 用户名 | 与 `--db-username` 相同 |
| `--os-password` | SSH 密码 | 与 `--db-password` 相同 |
| `--os-ssh-key-path` | SSH 私钥路径 | 无 |

### 6.1.4 采集控制参数
| 参数 | 说明 | 默认值 |
|---|---|---|
| `--os-collect-interval` | OS 采集间隔（秒），`0` 表示单次 | `0` |
| `--os-collect-duration` | OS 采集总时长（秒） | `0` |
| `--os-collect-count` | OS 采样点数量 | `0` |

### 6.1.5 参数互斥与约束（必须）
1. `--os-skip` 与 `--os-only` 互斥。
2. `--os-skip` 与任意 OS 采样控制参数互斥。
3. `--local` 与 SSH 参数互斥。
4. `--os-collect-duration` 与 `--os-collect-count` 互斥。
5. `--os-collect-interval > 0` 时，必须且只能搭配 `duration` 或 `count` 之一。
6. `--os-only` 模式下，DB 连接参数可省略。
7. 任何历史 Oracle 专用输入均应立即报错并输出迁移提示。

### 6.1.6 输出约束
1. 主链路必须产出：`collector.log`、`manifest.json`、`result.json`。
2. `run_id` 必须全局唯一，格式：`<db_type>-<host>-<yyyymmddThhmmssZ>`。
3. `--os-only` 为旁路模式：不进入主链路产物约束。

## 6.2 Analyzer CLI

### 6.2.1 基本语法
```bash
python3 analyzer/cli/db_analyzer.py --manifest <manifest.json> --result <result.json> --rule <rule.json> --out <summary.json> [选项]
```

### 6.2.2 参数表
| 参数 | 说明 | 默认值 |
|---|---|---|
| `--manifest` | manifest 文件路径 | 必填 |
| `--result` | result 文件路径 | 必填 |
| `--rule` | rule 文件路径 | 必填 |
| `--out` | summary 输出路径 | 必填 |
| `--log-level` | 日志级别 | `INFO` |
| `--strict-schema` | 强制 schema 校验可执行 | `false`（CI 必须 `true`） |

### 6.2.3 输入校验顺序（必须）
1. 参数解析与路径存在性检查。
2. JSON 反序列化检查。
3. schema 校验（`--strict-schema` 时必须可执行）。
4. 跨文件一致性校验。
5. 规则版本与 DB 类型校验。
6. 规则判定与 summary 生成。

### 6.2.4 失败中断与输出原子性（必须）
1. 任一校验失败立即中断，返回非 0 退出码。
2. 输出采用原子写：先写临时文件，再 `rename` 到目标路径。
3. 写入失败不得产出半成品 `summary.json`。

## 6.3 Reporter CLI

### 6.3.1 基本语法
```bash
db-reporter --run-dir <runs/<run_id>> [选项]
```

### 6.3.2 参数表
| 参数 | 说明 | 默认值 |
|---|---|---|
| `--run-dir` | 采集输出目录，必须包含 `manifest.json` 与 `result.json` | 必填 |
| `--out-docx` | 报告输出路径 | `<run-dir>/report.docx` |
| `--out-md` | 可选 Markdown 输出路径 | 不生成 |
| `--template-file` | 模板文件路径 | 内置默认模板 |
| `--rule-file` | 规则文件路径 | 内置默认规则 |
| `--python-bin` | Python 可执行文件 | `python3` |

### 6.3.3 职责硬约束（必须）
1. Reporter 禁止执行阈值判定与规则计算。
2. Reporter 仅根据 `result + summary` 渲染展示。
3. 输入契约不合法时必须失败退出，不得静默降级。

## 7. 错误码与失败语义

### 7.1 Collector 退出码（固定）
| 退出码 | 含义 |
|---|---|
| `0` | 采集成功 |
| `10` | 部分成功，可继续分析 |
| `20` | 采集失败，无有效结果 |
| `30` | 参数/连接/权限前置失败 |

### 7.2 Analyzer 退出码（40-49）
| 退出码 | 含义 |
|---|---|
| `0` | 成功 |
| `40` | 参数错误 |
| `41` | 输入文件不存在/不可读 |
| `42` | schema 校验失败 |
| `43` | 跨文件一致性校验失败 |
| `44` | 规则解析或判定失败 |
| `45` | 输出写入失败 |
| `49` | 未分类内部错误 |

### 7.3 Reporter 退出码（50-59）
| 退出码 | 含义 |
|---|---|
| `0` | 成功 |
| `50` | 参数错误 |
| `51` | 输入文件不存在/不可读 |
| `52` | 输入契约无效 |
| `53` | 模板文件无效 |
| `54` | 渲染失败 |
| `55` | 输出写入失败 |
| `59` | 未分类内部错误 |

### 7.4 失败语义联动（必须）
1. `manifest.exit_code in {20,30}` 时，`summary.failure` 必填。
2. 失败场景下 `summary.overall_risk` 必须为 `high`。
3. 失败场景下 `summary.abnormal_items` 必须为空。

## 9. 开发规范（强制）

## 9.1 代码质量硬限制（必须）
1. 函数长度 <= 50 行（不含空行）。
2. 文件长度 <= 300 行（超限必须拆分）。
3. 嵌套深度 <= 3。
4. 位置参数 <= 3，超出使用配置对象。
5. 圈复杂度 <= 10。
6. 禁止魔法数字，必须提取命名常量。

## 9.2 设计约束（必须）
1. 依赖注入：业务层不得硬编码具体实现。
2. 不可变优先：禁止在函数内修改输入参数。
3. 禁止静默降级：失败必须可见、可追踪。
4. 删除废弃兼容路径时必须同步更新文档与测试。

## 9.3 安全规范（必须）
1. 禁止明文写入任何密钥/凭据。
2. 日志必须对敏感字段脱敏。
3. 数据库访问必须参数化。
4. 外部输入必须在边界层完成校验与清洗。

## 9.4 日志规范（必须）
1. 日志分层：调试日志、业务摘要、错误栈。
2. 摘要日志必须包含：`run_id`、阶段、耗时、产物路径。
3. 错误日志必须包含：错误类别、上下文、建议动作。

## 9.5 配置规范（必须）
1. 优先级：CLI 参数 > 环境变量 > 默认值。
2. 冲突项必须报错退出，不得自动猜测。
3. 默认值必须在 CLI 文档中显式列出。

## 10. 测试规范与 CI 门禁

## 10.1 测试分层
1. 单元测试：函数逻辑、参数校验、映射与格式化。
2. 集成测试：组件内部流程（文件读写、schema、模板渲染）。
3. E2E 测试：采集 -> 分析 -> 报告全链路回放。

## 10.2 强制测试用例
1. CLI 参数矩阵：必填、互斥、默认值、非法组合。
2. 契约一致性：schema + 跨文件关联 + 计数守恒。
3. 缺失语义映射：`failed/skipped/not_applicable`。
4. 失败摘要：`exit_code=20/30` 的强约束。
5. Reporter 职责边界：禁止规则判定。

## 10.3 量化门禁（必须）
1. 单元测试覆盖率 >= 80%。
2. 后端单元测试超时硬限制：60 秒。
3. CI 必须开启 `--strict-schema`。
4. 任一门禁失败，流水线必须失败。

## 10.4 CI 推荐流水线
1. 静态检查（lint/format/type）。
2. 单元测试（覆盖率 + timeout）。
3. 契约校验（strict schema）。
4. E2E 冒烟（最小回放样例）。
5. 构建与发布。

## 10.5 推荐校验命令
```bash
# 契约校验（严格）
python3 tasks/validate_frozen_contracts.py \
  --manifest /path/to/manifest.json \
  --result /path/to/result.json \
  --summary /path/to/summary.json \
  --rule /path/to/rules/mysql/rule.json \
  --strict-schema

# 采样分析
python3 tasks/analyze_frozen_contracts.py \
  --manifest /path/to/manifest.json \
  --result /path/to/result.json \
  --rule /path/to/rules/mysql/rule.json \
  --out /path/to/summary.json
```

## 11. 实施任务拆解（工作包、依赖、DoD）

## 11.1 工作包 A：项目骨架与模块边界
- 输入：`docs/specs/db-check-contracts-prd.md`、`contracts/*`
- 输出：目录骨架、模块责任文档、基础脚手架
- 依赖：无
- DoD：
1. 三组件目录结构完成。
2. 模块边界与 import 方向检查规则可执行。
3. 规则目录固定为 `rules/mysql/rule.json`。
- 验收命令：
```bash
rg -n "analyzer/evaluator|reporter" docs/specs/开发规范.md
```

## 11.2 工作包 B：Collector MySQL CLI 与产物输出
- 输入：工作包 A 产物
- 输出：Collector CLI、参数校验、`manifest/result/log` 输出
- 依赖：A
- DoD：
1. `--db-type` 仅允许 `mysql`。
2. `--os-only` 旁路模式生效且不要求 DB 参数。
3. 互斥规则全部通过测试。
4. 主链路产物完整写入 `runs/<run_id>/`。
- 验收命令：
```bash
db-collector --db-type mysql --db-host 127.0.0.1 --db-port 3306 --db-username u --db-password p --dbname test --output-dir ./runs
```

## 11.3 工作包 C：Analyzer 判定引擎与 summary 生成
- 输入：Collector 产物 + `rules/mysql/rule.json`
- 输出：`summary.json`
- 依赖：A, B
- DoD：
1. schema 与跨文件一致性校验完整。
2. 缺失语义映射与计数守恒通过测试。
3. `40-49` 退出码落地并覆盖关键失败路径。
4. 输出写入采用原子策略。
- 验收命令：
```bash
python3 analyzer/cli/db_analyzer.py --manifest /path/to/manifest.json --result /path/to/result.json --rule /path/to/rules/mysql/rule.json --out /path/to/summary.json --strict-schema
```

## 11.4 工作包 D：Reporter 渲染链路与模板集成
- 输入：`runs/<run_id>/`
- 输出：`report.docx`
- 依赖：A, C
- DoD：

```bash
db-reporter --run-dir /path/to/runs/<run_id>
```
